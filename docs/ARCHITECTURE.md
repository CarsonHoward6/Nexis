# SnapRelay — Architecture Reference

Quick reference for every architectural decision. Use this during your demo walkthrough.

---

## System Overview

SnapRelay is a serverless file storage and sharing service. Users authenticate with Cognito, upload photos directly to S3 using pre-signed URLs, and access a shared gallery organized by group. File processing happens asynchronously via SQS.

---

## AWS Services Used (6/6)

| # | Service | Category | Why This One |
|---|---------|----------|--------------|
| 1 | **S3** | Storage | Industry standard for binary file storage. Pre-signed URLs enable direct browser-to-S3 uploads — no Lambda handles binary data. |
| 2 | **Lambda** | Compute | Serverless — no EC2 to manage. Each function has one job. Scales to zero when not in use (cheap for a student project). |
| 3 | **API Gateway** | Networking | Fronts all Lambdas with a single HTTPS endpoint. Cognito Authorizer validates JWTs before Lambda even runs. |
| 4 | **Cognito** | Auth | Managed auth — handles user pools, JWTs, token refresh, and group membership. No auth code to write or secure. |
| 5 | **DynamoDB** | Database | File metadata fits a key-value model perfectly. GSI lets us query by group without a full scan. No SQL, no connection pool. |
| 6 | **SQS** | Messaging | Decouples upload from processing. User gets instant response; thumbnail generation/status update happens asynchronously. |

---

## Data Flow — Upload

```
1. User selects file in browser
2. Browser → POST /upload/presign → API Gateway → Cognito validates JWT → presign Lambda
3. presign Lambda generates S3 pre-signed PUT URL (1hr expiry) → returns to browser
4. Browser → PUT file directly to S3 (not through Lambda)
5. Browser → POST /files → savePhoto Lambda → writes metadata to DynamoDB (status: "processing")
6. S3 ObjectCreated event → SQS message queued
7. processUpload Lambda polls SQS → updates DynamoDB (status: "ready")
8. Browser polls GET /files → sees status change → shows green "ready" badge
```

**Why direct-to-S3 upload?**
Lambda has a 6MB payload limit and 512MB /tmp storage. A Canon R6 RAW file is 25MB+. Routing uploads through Lambda would fail. Pre-signed URLs bypass this entirely.

---

## Data Flow — Download / Share

```
1. User clicks Download or Share on a file
2. Browser → POST /files/{id}/share → shareLink Lambda
3. Lambda queries DynamoDB for s3Key
4. Lambda calls S3 GetObjectCommand with the key → getSignedUrl (expires in 1hr for download, 24hr for share)
5. Signed URL returned to browser
6. Browser opens URL → S3 serves file directly to user
7. Share link works for anyone — no login required — until it expires
```

## DynamoDB Access Patterns

| Query | Index Used | Key Condition |
|---|---|---|
| My files | Main table | `userId = :uid` |
| Group gallery | GSI `groupId-uploadedAt-index` | `groupId = :gid` |
| Single file | Main table | `userId = :uid AND fileId = :fid` |

The GSI is what makes group galleries fast. Without it, you'd need a full table scan ($$$).

---

## Security Model

**Authentication:** Cognito User Pool issues JWTs on login. Tokens expire in 1 hour. Refresh tokens valid 30 days.

**Authorization:** API Gateway Cognito Authorizer rejects any request with an invalid or missing JWT before Lambda runs. Lambda extracts `userId` from the JWT claims — users can never access another user's files by guessing a fileId.

**File Access:**
- Private files: Only accessible via short-lived pre-signed URLs generated by authenticated Lambda
- Public files: Pre-signed URLs with longer expiry (24hr for share links)
- S3 bucket itself: Fully private — no direct public access to any object

---

## Why Not These Services?

**Not EC2:** Lambda is stateless and scales automatically. EC2 requires patching, scaling groups, and costs money 24/7 even at zero traffic.

**Not RDS:** File metadata doesn't need relational joins. DynamoDB's access patterns (get by userId, filter by groupId) are a perfect key-value fit. RDS would add VPC complexity and connection management.

**Not ElastiCache:** Would make sense if the gallery had millions of reads/second. For this project, DynamoDB's read capacity is sufficient. Could add ElastiCache as a read-through cache on `listFiles` if needed.

## Cost at Scale (Reference for Demo)

| Usage Level | Monthly Cost |
|---|---|
| Student project (current) | ~$0–2 |
| 100 active users, 10GB storage | ~$5–10 |
| 1,000 users, 100GB, 1M API calls | ~$30–50 |
| Enterprise (Wyre competitor) | ~$200+ |

S3 storage costs $0.023/GB. CloudFront egress is $0.085/GB. Lambda first 1M requests free every month. This stack is extremely cost-efficient.
